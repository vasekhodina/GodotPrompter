# GodotPrompter v1.5.0 — Tooling + Depth Pass Design

**Status:** approved (brainstorming)
**Date:** 2026-04-30
**Predecessor:** v1.4.1 (superpowers coexistence hotfix), v1.4.0 (Godot 4.3–4.5 source alignment)
**Sequencing:** Approach 1 — tooling lands first, then drives the depth pass

---

## 1. Goal

Ship v1.5.0 as a coordinated release that combines:

1. A **depth pass** on existing skills (Godot 4.5/4.6 source alignment + cross-references audit)
2. **Tooling/automation** that makes that depth pass and future releases mechanical (skill validator + marketplace release workflow)
3. **Two new agents** that fill specialist gaps (`godot-shader-author`, `godot-performance-profiler`)

No new skills. The skill count stays at 44.

The tooling lands first so the depth pass can be driven by validator output, and so the v1.5.0 release itself is the first dogfood of the marketplace automation.

---

## 2. Scope

### In scope

- All 44 existing skills reviewed for Godot 4.5/4.6 deltas; updated where genuine new API exists
- Cross-references audit across all 44 skills + 3 existing agents
- Two new agents under `agents/`
- Two repo-level scripts under `scripts/`
- One GitHub Actions workflow under `.github/workflows/`
- CHANGELOG, version bumps in 4 files, marketplace PRs

### Out of scope

- New skills (deferred — none picked for this release)
- Agent integration test plan expansion
- New platform support
- GDExtension / native skill (deferred)

### Versioning policy

- Repo minimum stays at **Godot 4.3+**
- New 4.5/4.6 content is **additive** with explicit version annotations: `### LookAtModifier3D (Godot 4.5+)`
- Where a 4.5+ idiom replaces a 4.3 one, both stay; the 4.3 form gets a deprecation note pointing at the new form
- Skills with no 4.5+ deltas in their domain are **not edited** — content is not manufactured

### 3.2 Tooling components

#### `scripts/validate-skills.mjs`

Node ES module (the repo's `package.json` already declares `"type": "module"`). No external dependencies required beyond what Node ships — reads files with `node:fs`, parses YAML frontmatter with a small inline parser (frontmatter is constrained enough that we don't need a YAML lib).

**Checks performed:**

| Severity | Rule |
|----------|------|
| error | Missing or unparseable YAML frontmatter |
| error | Missing required frontmatter keys: `name`, `description` |
| error | Cross-reference does not resolve to a real `skills/<name>/` folder. Sources scanned: (1) the "Related skills" line, (2) explicit `see **<name>** skill` / `(see **<name>** skill)` phrasings. Bare `**word**` emphasis is not parsed as a cross-reference — too noisy (e.g., `**collision_mask**`). |
| warning | Missing "Related skills" line in expected position (after H1, before first numbered section) |
| warning | Numbered section contains a GDScript code block with no paired C# block in the same section |
| warning | Missing implementation checklist at end of SKILL.md |

**Output modes:**
- Default: human-readable report grouped by skill
- `--json`: machine-readable for CI consumption

**Exit codes:** 1 if any errors, 0 otherwise (warnings don't fail the run).

**Targets:** `skills/*/SKILL.md` plus `agents/*.md` (cross-references only).

#### `scripts/bump-version.mjs`

Local helper — bumps the version string in:
- `package.json`
- `.claude-plugin/plugin.json`

Verifies they match before the bump, errors out if drift is detected.

For the two marketplace files (`skillsmith/.claude-plugin/marketplace.json` and `godot-prompter-marketplace/.claude-plugin/marketplace.json`):
- If sibling repos exist on disk (auto-detected at `../skillsmith` and `../godot-prompter-marketplace`), bumps them too
- Otherwise prints clear instructions for manual bumps with the exact paths to edit

Usage: `node scripts/bump-version.mjs 1.5.0`

#### `.github/workflows/release.yml`

Triggered on tag push matching `v*.*.*`. Runs on `ubuntu-latest`.

**Job 1 — verify:**
1. Checkout
2. Read version from `package.json`
3. Compare with tag (strip `v` prefix); fail if mismatch
4. Read version from `.claude-plugin/plugin.json`; fail if mismatch
5. Run `node scripts/validate-skills.mjs`; fail on errors

**Job 2 — release** (depends on verify):
6. Extract CHANGELOG section for this version
7. Create GitHub release with that body, marking as latest

**Job 3 — marketplace-prs** (depends on release; failure is non-fatal):
8. Use `MARKETPLACE_TOKEN` repo secret (PAT or fine-grained App token with `pull_requests:write` on both marketplaces)
9. Clone `skillsmith`, bump version in `.claude-plugin/marketplace.json`, open PR
10. Clone `godot-prompter-marketplace`, bump version in `.claude-plugin/marketplace.json`, open PR
11. If `MARKETPLACE_TOKEN` is unset: log a clear actionable message ("set MARKETPLACE_TOKEN secret or run the manual bumps from CONTRIBUTING.md") and exit 0 — release succeeds, PRs are user's manual responsibility for that release

**Failure handling:** if Job 1 or Job 2 fails, the workflow fails loudly and the user falls back to the manual release path from `CONTRIBUTING.md`. Job 3's failure does not block the release.

### 3.3 New agents

Both follow the existing `agents/<name>.md` format:

```yaml
---
name: <agent-name>
description: |
  When to use, with <example> blocks.
model: inherit
---
```

#### `godot-shader-author`

**Purpose:** specialist for shader work — Godot shader language, visual shaders, post-processing, Compositor effects.

**Triggers:** user requests a custom shader, post-processing effect, screen-space effect, custom material, visual shader graph, or Compositor effect.

**Anchor skills:** `shader-basics` (primary); `particles-vfx`, `2d-essentials`, `3d-essentials` (secondary).

**Distinguishing behavior:**
- Knows when to reach for visual shaders vs. text shaders
- Picks the right shader type: `shader_type canvas_item` vs. `spatial` vs. `particles` vs. `sky` vs. `fog`
- Calls out perf cost (overdraw, samples, branching) inline
- Always provides both the shader source and example usage in a `ShaderMaterial`

#### `godot-performance-profiler`

**Purpose:** diagnose performance issues in a user's project. Reads code, suggests profiler captures to take, interprets results, prescribes fixes.

**Triggers:** user reports lag, stutter, frame drops, draw call spikes, GC pauses (C#), physics slowdowns.

**Anchor skills:** `godot-optimization` (primary); relevant subsystem skill secondary depending on bottleneck (e.g., `physics-system`, `shader-basics`, `csharp-godot`).

**Distinguishing behavior:**
- Asks for profiler data before guessing — never optimizes blind
- Bottleneck taxonomy first: CPU vs. GPU, draw calls vs. fillrate, physics vs. scripts, GC
- Output structure: **bottleneck identified → fix per skill → verification steps**

Both agents are added to the `using-godot-prompter` skill index, and tool mappings in `skills/using-godot-prompter/references/` are updated where applicable.

### 3.4 Cross-references audit

The audit is **driven by** `validate-skills.mjs` output, not by manual reading.

**Procedure:**
1. Run validator → produces a list of broken cross-references
2. For each broken reference, classify:
   - **Skill renamed** → update the link
   - **Skill removed/never existed** → remove the reference
   - **Bidirectional mismatch** (A→B exists, B→A missing) → add the reciprocal where genuinely related; otherwise leave one-directional intentionally and document in the validator config (or skip the warning per-pair)
3. Re-run validator until clean

The audit also covers cross-references in `agents/*.md`.

### 3.5 Godot 4.5/4.6 source alignment

**Approach** (mirrors v1.4.0's source-alignment release, scoped to 4.5/4.6 deltas):

For each of the 44 skills:
1. Identify the skill's Godot subsystem(s)
2. Scan Godot 4.5 release notes and 4.6 in-development notes for additions to that subsystem
3. If genuine new API/feature exists, add an annotated section
4. If nothing material, skip — no edit

**Skills most likely to gain content** (preliminary; refined during execution):

- `physics-system` — Jolt updates, joint additions, soft body improvements
- `animation-system` — AnimationTree additions in 4.5/4.6
- `shader-basics` — Compositor enhancements, new built-ins
- `2d-essentials`, `3d-essentials` — new rendering features
- `multiplayer-basics`, `multiplayer-sync` — networking changes
- `xr-development` — OpenXR runtime updates
- `gdscript-patterns` — new language features (typed dictionaries, etc.)
- `csharp-godot` — GodotSharp API additions

**Validation gate per skill:** every newly added code example must be cited (in the implementation plan, not in the SKILL.md) to a Godot doc URL or merged PR. No speculative API.

## 4. Build Sequence

### Phase 1 — Tooling

1. Implement `scripts/validate-skills.mjs`
2. Implement `scripts/bump-version.mjs`
3. Implement `.github/workflows/release.yml`
4. Run validator against current `main`; if errors surface, that becomes the pre-work for Phase 2
5. Land Phase 1 in one PR (or split scripts vs. workflow if reviewing gets large)

### Phase 2 — Cross-references audit

6. Run validator → consume report
7. Fix broken cross-references in batches grouped by topic (e.g., "architecture-skill refs", "rendering refs")
8. Re-run validator until clean
9. One PR per batch

### Phase 3 — Godot 4.5/4.6 alignment

10. For each skill cluster (rendering, physics, scripting, networking, …), survey 4.5/4.6 deltas
11. Add annotated sections to skills with genuine deltas
12. One PR per skill cluster, with cited sources in PR description

### Phase 4 — New agents

13. Author `agents/godot-shader-author.md`
14. Author `agents/godot-performance-profiler.md`
15. Update `skills/using-godot-prompter/SKILL.md` index
16. Update tool-mapping references where they list agents
17. Land in one PR

### Phase 5 — Release

18. `node scripts/bump-version.mjs 1.5.0`
19. Update `CHANGELOG.md` with the v1.5.0 section
20. Commit, tag `v1.5.0`, push tag
21. Workflow runs → release created → marketplace PRs opened
22. Merge marketplace PRs

## 5. Validation

### Per-PR

- `node scripts/validate-skills.mjs` runs clean (no errors)
- Code examples manually compiled against Godot 4.5 (manual step per CLAUDE.md)
- C# parity preserved (or absence justified)
- CHANGELOG entry drafted

### Pre-tag

- All four pillars merged
- Repo-wide validator run is clean
- Both `package.json` and `.claude-plugin/plugin.json` show `1.5.0`
- CHANGELOG entry final

### Post-tag

- Workflow run succeeds
- GitHub release exists at `v1.5.0`
- Marketplace PRs open against both repos (or workflow logged a clear "no token" message)

## 6. Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Validator flags many existing issues, blocking Phase 1 | Land validator with rules permissive enough that current `main` passes; tighten rules in Phase 2 alongside fixes |
| `MARKETPLACE_TOKEN` not set up in time for v1.5.0 release | Workflow degrades gracefully with a clear error; user runs manual marketplace bumps from `CONTRIBUTING.md` for v1.5.0 itself; fix in v1.5.1 |
| Workflow has a bug exposed only on real tag push | Manual fallback path in `CONTRIBUTING.md` is preserved; v1.5.0 release proceeds manually if needed; workflow fix becomes v1.5.1 |
| 4.5/4.6 alignment pass scope balloons | Phase boundaries are per-skill-cluster PR; cut at any cluster boundary and defer rest to v1.5.1 |
| New agent triggers overlap with existing agents | Each agent definition explicitly documents when *not* to invoke it (i.e., when one of the existing three is more appropriate) |
| Bidirectional cross-ref churn (A↔B mismatch warnings flap) | Validator config supports an opt-out list for intentionally-asymmetric pairs |
