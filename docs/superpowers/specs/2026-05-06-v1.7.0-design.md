# GodotPrompter v1.7.0 Design

> **For agentic workers:** This is the design spec. The implementation plan lives at `docs/superpowers/plans/2026-05-06-v1.7.0-implementation.md` (written after design approval via `superpowers:writing-plans`).

## Goal

Establish a **token budget for skills** with measurement and enforcement, restructure the top-10 heaviest skills to fit it, expand GDScript depth via a new advanced skill, and add the next specialist agent (`godot-tools-engineer`) which dogfoods itself by closing its own skill's C# parity debt.

## Theme

**v1.7.0 is the token-budget release.** The budget rule, the restructured skills, and the new content (which itself fits the budget) are evidence of the same theme.

## Locked decisions (from brainstorming)

| Decision | Choice | Rationale |
|---|---|---|
| Token consumption focus | Measurement **and** reduction | Users need both — the number to plan around, and lower numbers to plan with |
| Restructure scope | Top-10 (skills ≥ 28 KB) restructured in v1.7.0; remaining 25 tracked as debt for v1.7.x | One-release pace matches v1.6.0; bigger sweep risks shallow per-skill work |
| Restructure pattern | Pattern X — core SKILL.md + `references/<topic>.md` | Preserves single-language discoverability; the natural cut point at "core + recipes" |
| Validator threshold | 16 KB warning (aspirational rule); 14 KB post-restructure target (headroom) | 16 KB is the natural break point; 14 KB target leaves room for incremental additions |
| Counting method | Bytes for validator + real tokenizer for docs page | Validator must run dependency-free in CI; docs need accuracy |
| GDScript depth shape | New `gdscript-advanced` skill (8 sections, GDScript-only by design) | Clean audience split (intermediate vs advanced); avoids bloating `gdscript-patterns` |
| New agent scope | `godot-tools-engineer` — editor-plugin scope only | GDExtension needs its own skill; deferred to v1.8 |
| Dogfood | Agent closes its own primary skill's parity (4 `addon-development` sections) | Continues v1.6.0 dogfood pattern |
| `addon-development` restructure | Restructured during dogfood (option B), not deferred | Stronger story: agent's first task showcases the full v1.7.0 toolkit |
| Release packaging | One v1.7.0 tag (not split into v1.7.0 + v1.7.1) | Cleaner narrative; v1.6.0 cadence already accommodates 10+ commits |
| GDExtension | Deferred to v1.8 | Multi-week authoring effort; cramming with token-budget sweep is too much |

## Architecture overview

**~16 commits across 5 phases.** Each commit independently validates and is independently revertable.

```
Phase 1 — Measurement infrastructure (3 commits)
Phase 2 — Top-10 skill restructure using Pattern X (10 commits)
Phase 3 — New gdscript-advanced skill (1 commit)
Phase 4 — godot-tools-engineer agent + addon parity dogfood (6 commits)
Phase 5 — Release plumbing (3 commits)
```

## Phase 1 — Measurement infrastructure

### `scripts/count-tokens.mjs`

Standalone Node 20+ script with two modes.

**Default mode:** `node scripts/count-tokens.mjs`
- Walks `skills/**/SKILL.md`, `skills/**/references/*.md`, `agents/*.md`.
- Per file: byte count, estimated token count (bytes ÷ 4), warning flag if SKILL.md ≥ 16 KB.
- Output: sorted table by token estimate (heaviest first), then summary line.
- Zero npm dependencies.

**Tokenizer mode:** `node scripts/count-tokens.mjs --tokenizer`
- Imports `@anthropic-ai/tokenizer` (Claude) and `js-tiktoken` (GPT cl100k_base, also used as a Gemini-Code proxy).
- Per file: real Claude tokens, real GPT tokens, byte count.
- These two packages enter `package.json` as **`optionalDependencies`** so CI without them still works.

**Markdown output:** `node scripts/count-tokens.mjs --tokenizer --markdown` produces the table that ships in `docs/token-budget.md`.

**JSON output:** `node scripts/count-tokens.mjs --json` for machine consumption.

### Validator rule (in `scripts/validate-skills.mjs`)

Add one rule with severity **warning** (same class as C# parity warnings):

```
[token-budget-exceeded] skills/<name>/SKILL.md is <size> KB (≥ 16 KB)
```

The 16 KB threshold is a tunable constant at the top of the script.

The validator also gains an **orphan check** for the new `references/` pattern: every `skills/<name>/references/*.md` file must be linked from its parent `skills/<name>/SKILL.md` via a relative link. Reference files that no SKILL.md links to emit a `[orphan-reference]` warning.

### Validator warning baseline (release target)

| Category | v1.6.0 | v1.7.0 release |
|---|---|---|
| C# parity (deferred) | 23 | 19 (−4 from `addon-development` closures) |
| Accepted GDScript-only | 9 | 16 (+7 from `gdscript-advanced` sections 2–8; section 1 is framing only, no code block) |
| Token-budget exceeded | 0 | 24 (top-10 + `addon-development` cleared; 24 over-budget remaining) |
| **Total warnings** | **32** | **~59** |
| **Errors** | **0** | **0** |

The warning total *grows* this release because the new rule surfaces existing debt that was previously invisible. CHANGELOG framing makes this explicit.

### `docs/token-budget.md`

Three sections:

1. **What this is** — One paragraph: the budget rule, why it exists.
2. **Current costs** — Auto-generated table (manually regenerated each release) listing all 45 skills + 9 agents with: byte count, Claude tokens, GPT tokens, status (under-budget / over-budget / restructured-in-vX). Generated by `node scripts/count-tokens.mjs --tokenizer --markdown`.
3. **How to read this** — Practical guidance: a typical agent loads 1 primary skill + 2-3 cross-referenced skills; show what that costs at p50 / p90 across the catalog.

Committed as a static snapshot. Regeneration step added to `CONTRIBUTING.md` release checklist.

### `docs/superpowers/notes/2026-05-06-token-budget-debt.md`

New tracking note. Mirrors `2026-04-30-csharp-parity-debt.md` structure:

- **v1.7.0 progress** — 11 skills closed (top-10 + `addon-development`)
- **Deferred (count: 24)** — Each remaining over-budget skill listed with current size and a one-line strategy hint
- **Accepted** — Empty initially

Future v1.7.x releases pick rows off this list, identical pattern to v1.6.x C# parity work.

## Phase 2 — Top-10 skill restructure (Pattern X)

### Pattern X: the cut rule

For each skill ≥ 28 KB, SKILL.md keeps:

1. YAML frontmatter
2. Title + intro paragraph
3. Related skills line
4. **Distinguishing-choices table** (the "when to use what" decision matrix)
5. **One canonical recipe per major topic**
6. Inline link out: `> See [references/<topic>.md](references/<topic>.md) for <variant>.`
7. Implementation checklist

What moves to `skills/<name>/references/<topic>.md`:

- Edge-case recipes (uncommon variants)
- Advanced patterns (production-grade extensions of the canonical recipe)
- Long code blocks where the *concept* fits in SKILL.md but the full implementation is bulky
- Per-engine-version annotations that would expand SKILL.md significantly

What stays in SKILL.md regardless of size:

- Anti-pattern callouts (short, high-value)
- C# parity blocks for the canonical recipe
- Cross-reference lines

### Per-skill cut targets

Order is heaviest-first. Each skill is its own commit.

| # | Skill | Current | Target | Strategy |
|---|---|---|---|---|
| 1 | `godot-optimization` | 37 KB | ~14 KB | Per-bottleneck deep dives → references (CPU, GPU, draw-calls, physics, GC each become a reference file) |
| 2 | `ai-navigation` | 37 KB | ~14 KB | Steering-behavior implementations + behavior-tree code → references |
| 3 | `animation-system` | 35 KB | ~14 KB | IK recipes (added v1.6.0) and BoneConstraint3D deep dive → references; solver-comparison table stays in core |
| 4 | `3d-essentials` | 35 KB | ~14 KB | Material/lighting/GI deep dives → references; distinguishing choices stay in core |
| 5 | `physics-system` | 34 KB | ~14 KB | SoftBody3D, Jolt-specific tuning, ragdoll recipes → references |
| 6 | `dedicated-server` | 32 KB | ~14 KB | Deployment recipes (Docker, systemd, lobby patterns) → references |
| 7 | `hud-system` | 29 KB | ~14 KB | Minimap, damage-numbers, notifications recipes → references |
| 8 | `inventory-system` | 29 KB | ~14 KB | Drag-drop UI, equipment slots, weight system → references |
| 9 | `audio-system` | 28 KB | ~14 KB | Bus routing, music management, SFX pooling deep dives → references |
| 10 | `godot-ui` | 28 KB | ~14 KB | Responsive recipes, FoldableContainer deep dive, theme inheritance → references |

The 14 KB target leaves headroom under the 16 KB warning. Each skill ends up with 2-5 reference files.

### Cross-reference rule

When a SKILL.md links to its own `references/` file, the link is **always relative** (`references/foo.md`, not absolute). Validator orphan check enforces every reference file is linked from its parent SKILL.md.

### Agent updates

After restructure, the following agents add one line to their process to mention reading deeper material from `references/`:

- `godot-game-architect` (most domain skills)
- `godot-game-dev` (most domain skills)
- `godot-performance-profiler` (`godot-optimization` is its primary skill)
- `godot-animator` (`animation-system` IK references)

The line takes the form: *"For deeper material on `<topic>`, also read `skills/<name>/references/<topic>.md`."*

## Phase 3 — `gdscript-advanced` skill

### Audience and positioning

`gdscript-patterns` (existing) covers **language fundamentals**. Audience: intermediate.

`gdscript-advanced` (new) covers **production-grade depth**. Audience: shipping a real game.

Both are **GDScript-only by design**. Validator accepts the C# parity warnings under the same allowlist as `gdscript-patterns`.

### Sections (8 total, ~14 KB SKILL.md)

1. **When to reach for advanced GDScript** — one-paragraph framing
2. **Performance idioms** — `static` vars/functions, `Vector2i` vs `Vector2`, `PackedArray*` vs `Array`, typed Dictionary access, `is_instance_valid` vs null check costs
3. **Metaprogramming** — `Callable.bind/call/call_deferred`, `Object.set/get/has_method`, when to use them vs typed code; security gotchas of `call(name)` from data
4. **`@tool` lifecycle** — Editor vs runtime branches, `Engine.is_editor_hint()` patterns, `notification(NOTIFICATION_*)` for editor events, common `@tool` bugs
5. **Async pitfalls** — `await` deadlocks, signal-vs-Callable for completion, `await` in `_ready()` ordering issues, `SceneTree.process_frame` patterns
6. **Signal vs Callable design choices** — when each wins, GC implications, decoupling patterns
7. **Profiler-driven idioms** — what shows up in the profiler and how to fix it (string concat in `_process`, allocator churn, deep node lookups, signal storms)
8. **Common pitfalls** — short list of "looks fine, isn't" patterns: lambda capture-by-reference, `@onready` ordering, static var lifecycle across scene reload, `Resource` sharing surprises

### References

- `skills/gdscript-advanced/references/tool-script-recipes.md` — full `@tool` patterns with editor-only logic
- `skills/gdscript-advanced/references/profiler-recipes.md` — annotated before/after for common bottlenecks
- `skills/gdscript-advanced/references/metaprogramming-recipes.md` — Callable patterns and security guidance

### Cross-references

- **Reciprocal** with `gdscript-patterns`
- **Listed by** `godot-optimization`, `godot-debugging`, `godot-performance-profiler` agent

### Validator

Added to validator's `gdscript-only-by-design` allowlist. C# parity warnings for its sections accepted.

## Phase 4 — `godot-tools-engineer` agent + dogfood

### Agent positioning

Specialist for **editor-side tooling**: `EditorPlugin`, `EditorInspectorPlugin`, `EditorImportPlugin`, custom inspectors, `EditorNode3DGizmoPlugin`, dock panels, `@tool` scripts that ship as plugins, plugin testing, plugin distribution. Both GDScript and C# (with `#if TOOLS` guards for C#).

GDExtension is **explicitly out of scope** (deferred to v1.8).

### Agent file structure (`agents/godot-tools-engineer.md`)

Same shape as the existing 8 specialists:

- YAML frontmatter with name, multi-line description, three `<example>` blocks
- **Your skills** — primary `addon-development`, supporting `gdscript-patterns` / `gdscript-advanced` / `csharp-godot` / `csharp-signals` / `godot-debugging`
- **Your process** — clarify (plugin vs `@tool` script?) → pick the right `Editor*Plugin` subclass → guard with `#if TOOLS` / `Engine.is_editor_hint()` → write GDScript first, C# parity → register in `plugin.cfg`
- **Distinguishing choices** — `EditorPlugin` vs `@tool` script; `EditorInspectorPlugin` vs `EditorProperty`; when to ship as plugin vs in-project; `addons/` layout
- **Output format** — plugin scaffold + at least one runnable file
- **When NOT to use this agent** — runtime gameplay code (use `godot-game-dev`), shaders (use `godot-shader-author`)

### Three trigger examples

1. **Custom inspector for a Resource** — "I want a custom inspector for my `ItemData` resource so designers can preview the item icon" → `EditorInspectorPlugin` work
2. **`@tool` script for a level designer** — "I need a `@tool` script that snaps my placed nodes to a grid in the editor" → `@tool` lifecycle work
3. **Editor gizmo for a custom node** — "How do I add a 3D gizmo for my custom `Spawner3D` node?" → `EditorNode3DGizmoPlugin` work

### Routing tweaks (existing agents)

Append one sentence to:

- `godot-game-dev`: *"For editor plugins, `@tool` scripts, custom inspectors, or gizmos, prefer `godot-tools-engineer`."*
- `godot-game-architect`: same line.

### The dogfood — restructure + 4 parity closures

The agent's first job is its own primary skill: restructure `addon-development` to fit the budget, then close its 4 deferred C# parity sections.

**Order:** restructure first (one commit), then four parity commits (one per section).

**Restructure target:** `addon-development` is currently 20 KB. Target: ~14 KB. Pattern X applied — long plugin scaffolds and `EditorImportPlugin` deep dives move to `references/`.

**Parity closures (one commit each):**

| Section | C# block content |
|---|---|
| **4. Custom Inspector Plugin** | `EditorInspectorPlugin` subclass with `#if TOOLS` guard, `_CanHandle`, `_ParseBegin`, `_ParseProperty` overrides, registration boilerplate in the `EditorPlugin` subclass |
| **6. Custom Resource Editors** | `EditorResourcePicker` embedded in a dock tool control, `EditorResourcePreviewGenerator` subclass with `_Handles` and `_Generate` overrides, `#if TOOLS` throughout |
| **7. Gizmos** | `EditorNode3DGizmoPlugin` subclass with `_Init`, `_GetGizmoName`, `_HasGizmo`, `_Redraw`, `_GetHandleValue`, `_SetHandle`, `_CommitHandle`; undo/redo wiring; `#if TOOLS` guards |
| **8. Testing Plugins** | C# equivalent of plugin reload snippet, `print()` → `GD.Print()` logging, lifecycle gotcha note for C# compilation failures |

After all four:

- Deferred C# parity warnings: 23 → 19
- `addon-development` becomes the **first skill with fully closed parity**

Each parity commit's C# block lands in whichever file (SKILL.md or `references/`) the section ended up in after restructure.

### Per-section commit shape

Identical to v1.6.0:

1. Open `addon-development/SKILL.md` (or applicable reference file), locate section
2. Add `csharp` block immediately after existing `gdscript` block
3. Run validator, confirm warning count drops by 1
4. Update `2026-04-30-csharp-parity-debt.md` with strikethrough + `(closed in v1.7.0)` annotation
5. Commit with `feat(skills): close C# parity for addon-development/<section>`

### Risks and mitigations

| Risk | Likelihood | Mitigation |
|---|---|---|
| Pattern X loses important context — readers skip linked references | Medium | SKILL.md keeps distinguishing-choices table + canonical recipe + anti-patterns. References are *deeper* recipes, not *required* reading. Validator orphan check enforces every reference is linked from SKILL.md. |
| 14 KB target unreachable for some top-10 skills | Medium | Target is aspirational; warning fires at 16 KB. Per-skill commit shape allows independent review. |
| New tokenizer dependencies break CI | Low | They're `optionalDependencies`. CI runs validator (default mode, byte-count only). Tokenizer mode is opt-in. |
| Warning count grows at release (+25 token-budget warnings) — bad optics | Low | CHANGELOG release note explicitly frames the budget rule as the reason. Headline number is still "0 errors". |
| `addon-development` restructure-during-dogfood collides — moving sections then adding C# blocks risks merge confusion | Medium | Two-step ordering enforced: (1) restructure commit lands first, (2) parity commits add C# blocks to whichever file each section ended up in. Each is reviewed independently. |
| Plan length — 16 commits in one branch | Low | v1.6.0 was 10; v1.7.0 is ~16. Each commit independently revertable. `superpowers:subagent-driven-development` handles this scale. |

### Out of scope (explicit)

- GDExtension skill / C++ native module authoring → v1.8
- Restructuring the remaining 24 over-budget skills → v1.7.x point releases
- Closing remaining 19 C# parity sections → v1.7.x and v1.8
- Auto-regeneration of `token-budget.md` in CI (manual for now)
- Godot 4.7 alignment (4.7 not yet shipped)

### Smoke tests at release

After tagging:

1. `node scripts/count-tokens.mjs` — verify output sane on a fresh checkout
2. `node scripts/count-tokens.mjs --tokenizer` after `npm install` — verify Claude/GPT columns populate
3. Open a Godot 4.5 project, invoke `godot-tools-engineer` with one of its three trigger examples — confirm it loads `addon-development/SKILL.md` and references the right section
4. Open `gdscript-advanced` SKILL.md in Claude Code with `Skill` tool — verify it loads cleanly

## Acceptance criteria

The release is complete when:

- [ ] `scripts/count-tokens.mjs` exists and runs in default + tokenizer modes
- [ ] Validator emits `[token-budget-exceeded]` warnings for skills ≥ 16 KB
- [ ] Validator emits orphan warnings for unreferenced `references/*.md` files
- [ ] `docs/token-budget.md` exists with current per-skill / per-agent table
- [ ] All 10 top skills (≥ 28 KB) restructured to ≤ 16 KB (target ~14 KB)
- [ ] `addon-development` restructured to ≤ 16 KB
- [ ] `gdscript-advanced/SKILL.md` exists, ≤ 16 KB, with 8 sections + 3 reference files
- [ ] `gdscript-advanced` accepted by validator's `gdscript-only-by-design` allowlist
- [ ] `agents/godot-tools-engineer.md` exists with three trigger examples
- [ ] `godot-game-dev` and `godot-game-architect` carry the routing note
- [ ] All 4 `addon-development` C# parity sections closed; parity debt note updated
- [ ] Token-budget debt note exists with 24 deferred skills listed
- [ ] README and CLAUDE.md show 9 agents / 45 skills
- [ ] CONTRIBUTING.md includes the token-budget regeneration step
- [ ] CHANGELOG has the v1.7.0 section

- [ ] Validator baseline at release: 0 errors, ~59 warnings (matches the breakdown above ± 2)
- [ ] Release tag `v1.7.0` pushed; GitHub release created; marketplace PRs opened (or manual fallback executed)
